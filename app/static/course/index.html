01 The button you clicked

You type a goal and hit Run. Then what?

🎯
Why care: this one path ties the whole app together. Once you can follow it, you can tell your AI assistant exactly which step to change instead of describing the whole feature.
🏃‍♀️
Metaphor · the relay race

Your goal is a baton. The Campaign page hands it to the runner, the runner hands it to each attack strategy, and the last leg hands the result to a judge. Nobody runs the whole track alone.

You · the page→ startCampaign()→ strategies→ the judge→ graded report🏁
app/src/lib/components/tools/campaign/CampaignTool.svelte
function doRun() {
  confirmPending = false;
  // snapshot the last report before this run replaces it
  if (campaignData && campaignRun?.status !== 'running') {
    pastReports = [campaignData, ...pastReports].slice(0, 3);
  }
  startCampaign({
    goal: goal.trim(),
    targetModel: targetPref.value,
    judgeModel: judgePref.value,
    strategies: selectedStrategies,
    vault: vaultStore
  });
}

In plain English

  1. You click Run → this function fires.
  2. It tucks away your previous result so you can compare runs.
  3. Then it hands everything off in one call: your goal, the target model, a separate judge model, the chosen attack strategies, and the saved-payloads vault.
  4. That's it — the page's job is done. The startCampaign engine takes over.
Spot it

You hit Run and… nothing. No error, no report appears. Where do you look first?

It's the provider. Cryptex is BYOK: the Run button stays disabled (and shows "Configure a provider") until you've added a key in Settings. doRun() never even fires. First rule of debugging: confirm the obvious gate before suspecting the code.
02 Meet the cast

Five players. Each does one job.

🧭
Why care: when you ask AI to "change how decoding works," it helps to know that lives in a transformer file — not the gateway, not the UI. Naming the right actor is half the instruction.
🎭
Metaphor · the theater troupe

Each actor plays exactly one role. And the casting call is open: drop a new actor's file into the right folder and the show finds them automatically — nobody updates a cast list.

TransformerI turn text into ciphers and back. There are 159 of us.
AI GatewayI'm the one phone line to every AI model. Everyone calls through me.
Campaign engineI take one goal and fan it across many of your attacks, then grade them.
ToolShellI'm the frame every tool page wears — title, status, history, errors.
StoresI remember your inputs in the browser and tell the screen when things change.
src/transformers/cipher/caesar.js · one actor
export default new BaseTransformer({
  name: 'Caesar Cipher',
  priority: 60,
  shift: 3,
  func: function(text, options) {
    const shift = options.shift ?? this.shift;
    return this._encode(text, shift);
  },
  reverse: function(text, options) { /* shift back */ },
  detector: function(text) { /* looks like letters? */ }
});

How the casting call works

  1. Every transformer is just a file that export default new BaseTransformer({…}) — a name, a func to encode, and an optional reverse.
  2. The registry uses a glob import to auto-find all of them:
// app/src/lib/transformers/registry.ts
const modules = import.meta.glob(
  '../../../../src/transformers/*/*.js',
  { eager: true }
);
You direct the AI

You want to add a brand-new cipher. What do you tell your AI assistant to do?

Just the one file. Because the registry globs the folder at build time, a new file is auto-discovered — there's no manifest to update and the UI lists it on its own. Knowing this saves you (and the AI) from editing files that don't need touching.
03 How they talk

One front door for every AI call.

🔌
Why care: "the wrong model answered" or "nothing came back" almost always traces to this one function. Knowing it exists means you check one place, not ten.
☎️
Metaphor · the switchboard

Every model has a name tag like openrouter:gpt-4o. The gateway reads the part before the colon and patches your call to the matching line. The colon-prefix is the routing.

openrouter:gpt-4o→ resolve()→ OpenRouter adapter→ the model 🤖
app/src/lib/ai/gateway.ts
const PREFIX_RE = /^(openrouter|anthropic|openai-compat):(.+)$/;

export function resolve(modelId) {
  const m = PREFIX_RE.exec(modelId);
  if (m) {
    const [, providerId, inner] = m;
    // …openai-compat: peel off the instance id…
    const record = listProviders()
      .find((p) => p.id === providerId);
    if (!record) throw new GatewayError(
      `Provider not configured: ${providerId}`);
    return { adapter: buildAdapter(record), modelId: inner };
  }
}

In plain English

  1. A model id is split by its colon: openrouter | gpt-4o.
  2. The first part says which provider; the rest is the real model name.
  3. It finds your configured provider for that prefix and builds its adapter.
  4. No matching provider? It throws "Provider not configured" — the exact error you'd see in the console.
Spot the cause

You picked an Anthropic model, but the run errors with Provider not configured: openrouter. Why?

It's the missing prefix. An unqualified id (no provider: in front) routes to OpenRouter by default. If you only configured Anthropic, that fallback fails. The fix Cryptex now ships: the model picker only shows models from providers you've actually configured.
04 The outside world

Your key stays home. The models live abroad.

🔐
Why care: two things confuse everyone — "where does my API key go?" and "why is the model list empty sometimes?" Both have the same root: Cryptex is a static site with no backend.
🔑
Metaphor · keys in your pocket, calling abroad

Your API key sits in your browser's localStorage — like keys in your pocket, never handed to a stranger. But fetching the model list is like calling a foreign number that may refuse your call: that refusal is CORS.

your browser→ OpenRouter /models ✓· blocked API ✗ → shipped list
app/src/lib/ai/catalog.svelte.ts
async function fetchAll(signal) {
  // only providers you turned on
  const providers = listProviders()
    .filter((p) => p.enabled);
  // …ask each one for its /models list…
}

// what the picker actually shows:
get list() {
  return items.length > 0 ? items : FALLBACK_MODELS;
}

In plain English

  1. Cryptex asks each enabled provider for its live model list.
  2. OpenRouter allows browser calls, so its list loads. Some APIs refuse (CORS) — so the picker shows a built-in FALLBACK list instead.
  3. In local dev a proxy hides this; in production (a static site) there's no proxy, which is why a custom endpoint can come up empty.
  4. Through all of it, your key never leaves the browser.
Scenario

Models load fine when you run Cryptex locally, but the list is empty in the deployed site for your custom endpoint. Why?

The proxy. The dev server proxies provider calls so CORS never bites; the static production build has no server to proxy through. For known providers there's a shipped fallback list — for a custom endpoint there isn't, so you type the model id by hand.
05 The clever tricks

How the screen reacts on its own.

📊
Why care: Cryptex is built on Svelte 5 runes. Understanding them lets you ask for UI changes precisely — and avoid the exact trap that took the app down (next module).
🧮
Metaphor · a spreadsheet

$state is a cell you type into. $derived is a formula cell that recalculates itself. $effect is an automatic action that fires when a cell it watches changes.

app/src/lib/stores/_persisted.svelte.ts
export function createPersistedState(key, initial) {
  let value = $state(initial);
  if (browser) {
    const raw = localStorage.getItem(key);
    if (raw !== null) value = JSON.parse(raw);
    $effect.root(() => {
      $effect(() => { localStorage.setItem(key, JSON.stringify(value)); });
    });
  }
  return {
    get value() { return value; },
    set value(next) {
      // dedup: skip equal writes (stops effect loops)
      if (JSON.stringify(next) === JSON.stringify(value)) return;
      value = next;
    }
  };
}

In plain English

  1. This makes a value that remembers itself across reloads (saved to localStorage).
  2. It's a $state cell, so any screen reading it updates when it changes.
  3. An $effect auto-saves it whenever it changes — you never write save code.
  4. That last dedup guard ("skip equal writes") is the hero of the next module.
Spot the danger

An $effect contains store.value = { ...store.value, goal }. What's risky?

Read-and-write of the same value = a loop. The effect depends on what it changes, so it never settles. Hold that thought — it's the real bug we trace next. (The dedup guard above is one of the two fixes.)
06 When things break

A real bug: the app that looked fine but was dead.

🩺
Why care: the most useful skill isn't writing code — it's reading a crash and finding the cause. This is a true story from this codebase, fixed days ago.
🐍
Metaphor · the ouroboros

The snake that eats its own tail. An $effect that depends on the very value it changes can never stop — it chases itself until the engine gives up and the whole page freezes.

effect → write →
re-trigger → write →
…💥

The page still rendered (the HTML was there) — but every button was dead, because the reactivity engine had crashed on load with effect_update_depth_exceeded.

CampaignTool.svelte · BEFORE (the bug)
let goal = $state(sharedContext.goal);
$effect(() => {
  // setter spreads (READS) the object it WRITES → loop
  sharedContext.goal = goal;
});
CampaignTool.svelte · AFTER (the fix)
$effect(() => {
  const g = goal;            // depend on `goal` only
  untrack(() => {            // read+write off the graph
    if (sharedContext.goal !== g) sharedContext.goal = g;
  });
});

The deeper lesson: it shipped because old checks only confirmed pages rendered — never that they were clickable. Cryptex now runs a mount test that actually flushes effects, so this class of bug fails the build instead of the user.

Debug it

An app renders perfectly, but every button is dead and the console shows effect_update_depth_exceeded. What kind of bug, and what do you check first?

A reactive loop. "Renders but nothing works + update-depth error" is the signature of an effect crashing the engine on load. The cause is almost always an effect that writes something it depends on. You just diagnosed a real production incident. 🎓
